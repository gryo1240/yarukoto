### アプリの特徴(記事の訴求ポイント)

1. **毎日手書きしていたToDoを自動化** — 「定番」を一度登録すれば、毎朝自動でその日のリストに並ぶ
2. **繰り返しは4パターン** — 毎日/曜日ごと(月・木のゴミ出し等)/週に○回(曜日は柔軟)/月に○回
3. **「週に○回」タスクは候補トレイに** — 「掃除機がけ 今週1/3」のように完了回数が見え、「やる」を押した日に入る
4. **時間決めがラク** — 朝・昼・夕方・夜のブロック+決まった予定だけ時刻指定。「病院 10:00」と打つだけで時刻を拾う
5. **完了したら下の「完了」リストへ移動** — やり残しが一目でわかる
6. **昨日の残りは翌朝「繰り越す/消す」を選べる**
7. **夫婦のスマホでリアルタイム同期(任意)** — 無料のFirebaseを使う。設定しなくても1台で完結して使える
8. 担当の色分け(ニックネーム2人+「2人で」)、「自分の分」フィルタつき

## 記事の下書き(そのまま使用・編集可)

タイトル案:
- 「毎日同じToDoを手書きしていませんか? 夫婦で共有できる無料アプリ『きょうのやること』」
- 「定番の家事が毎朝自動で並ぶ、家族のやることリストアプリ(無料)」

---

**毎日のやることを紙に手書きして、夫婦で「これ何時にやる?」と相談していませんか?**

うちも同じでした。しかも書く内容は毎日ほとんど同じ。その「毎日同じことを書く手間」をなくすために作られたのが、無料Webアプリ「きょうのやること」です。

**定番を一度登録すれば、毎朝勝手にリストができる**

洗濯・食器洗いのような毎日のこと、月・木のゴミ出しのような曜日のこと、「週に3回の掃除機がけ」のような柔軟なことまで、4パターンの繰り返しに対応。週○回のタスクは「今週1/3」と進み具合が見えるので、"そろそろやらなきゃ"がわかります。

**時間決めは「ざっくり」と「きっちり」の両方**

タスクは朝・昼・夕方・夜のブロックに並び、病院の予約など決まった予定だけ時刻を付けられます。夫婦それぞれのスマホで同期できるので(無料・任意設定)、2人でリストを見ながらその場で相談して決められます。

**終わったらチェック → 下の「完了」へ**

完了したものはリストの下にまとまるので、残りが一目瞭然。昨日やり残したことは、翌朝「繰り越す/消す」を選べます。

**インストール方法(無料・登録不要)**

- Android: [アプリを開く](https://gryo1240.github.io/yarukoto/) → Chromeメニュー →「アプリをインストール」
- iPhone: Safariで開く → 共有ボタン →「ホーム画面に追加」

データは端末の中(同期時は自分のFirebase)にだけ保存され、外部に送信されません。

## アプリ側を修正したい場合

- 本体は `index.html` 1ファイル。`main` に push で GitHub Pages に自動デプロイ
- 同期は Firebase Realtime Database(compat SDK をCDNから動的ロード)。利用者自身のconfigを設定タブに貼る方式で、DBパスは `rooms/{rid}`
- データ構造: `templates`(rule: "daily" | {days:[0-6]} | {flex:"week"|"month", n})、`days/{YYYY-MM-DD}/tasks`。テンプレ由来タスクのキーは `t_{tplId}` で冪等
- 「今日は外す」は `days/{d}/skip` に記録。**skipを書いてからタスクを消す順序を守る**こと(逆だと再描画で復活する)

## 同期機能のセキュリティ設計(公開可否の判断材料)

「Firebaseプロジェクトの所在」「セキュリティルール」の確認への回答。結論: **同期は一般公開してよい設計**(SYNC_UI_ENABLED は true に戻し済み)。

- **中央データベースは存在しない(BYO Firebase方式)**: アプリには開発者のAPIキーやデータベースは一切同梱されていない。同期を使いたい利用者だけが、自分のGoogleアカウントで自分専用のFirebaseプロジェクト(無料枠)を作り、そのconfigをアプリの設定タブに貼る。**データの所在は各利用者自身のプロジェクト**であり、公開者・開発者は誰のデータにもアクセスできず、ホスティングや管理の責任も負わない
- **セキュリティルール**: アプリ内の手順書(「きょうのやること」設定タブ)を2026-07-15に更新し、全開放(root read/write true)ではなく**ルームID単位でのみ読み書きできるルール**を案内するようにした:
  `rooms` / `closet` / `souji` の各 `$rid` 配下のみ read/write true。ルートや一覧の読み取りは不可 = ルームIDを知らない第三者は何も読めない
- **ルームID**: 招待コードに含まれる十数桁のランダム文字列で、事実上の合鍵。記事には「招待コードは家族以外に送らない」を明記すること
- **残るリスク(正直に)**: 招待コード(config+ルームID)が第三者に渡ると、その家族のルームは読み書きされうる。Googleログイン等の認証を伴わない「合鍵方式の簡易共有」であることを理解した人向けの任意機能、という位置づけで紹介するのが正確
- 同期を設定しない利用者は従来どおり端末内保存のみで、外部通信は発生しない
